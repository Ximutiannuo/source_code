# 专项计划（ahead-plan）高并发部署说明

适用于 **约 200 人同时** 使用专项计划管理页面的场景。

## 1. 已做的优化

- **连接池**：默认库 `pool_size=40`、`max_overflow=60`（单进程最多约 100 连接）；角色库也配置了 `pool_size`/`max_overflow`，避免默认过小。
- **批量更新 API**：`POST /api/ahead-plan/batch-update`。前端对「同一作业多周赋相同值」（作业描述、备注）改为一次批量请求，显著减少并发请求数与连接占用。
- **死锁**：优先**避免**死锁——批量审批、批量删除、批量更新在提交前均按 `id` 顺序执行 **SELECT FOR UPDATE** 加行锁，再修改/删除，使所有事务加锁顺序一致，从源头降低死锁概率；另保留死锁重试（MySQL 1213 时最多重试 3 次）作兜底。

## 2. MySQL 配置建议

总连接数 ≈ **Gunicorn workers × (pool_size + max_overflow)**。  
当前默认引擎每进程 40+60=100，若 `workers = 2*CPU+1`（例如 17），则理论峰值约 **1700**。  
实际 200 人同时操作时，请求会分散，但需保证 MySQL 不会因连接数拒绝请求。

建议在 MySQL 中设置：

```ini
[mysqld]
max_connections = 800
```

或按实际 worker 数计算：`max_connections >= workers * (pool_size + max_overflow)`，并留一定余量。  
修改后需重启 MySQL（或按数据库版本做在线调整）。

## 3. Gunicorn 与负载

- `gunicorn_config.py` 中 `workers = cpu_count() * 2 + 1`、`worker_connections = 1000` 已可支撑较多并发。
- 若单机 CPU 较多，可保持现状；若希望控制单机连接数，可适当减小 `backend/app/database.py` 中的 `pool_size`/`max_overflow`，并相应调低 MySQL `max_connections`。

## 4. 简要结论

- 每人操作内容不同、且多为「单格编辑」或「一次批量更新」时，**单条更新 + 批量更新** 已能明显降低连接与请求数，系统更易撑住 200 人。
- 确保 **MySQL max_connections** 不小于上述估算，并已应用本次后端的连接池与批量更新、死锁重试改动。
